# 临时源 pull 与全自动审核后写回设计

- 日期：2026-05-07
- 主题：为当前标定项目新增 `temp_path` 本地临时源 pull、GUI 全自动执行模式，以及“审核写回”与“后台 pull/move/upload”解耦
- 状态：已确认设计，待用户审阅书面 spec

## 1. 背景

当前项目已经具备一条可用但仍偏人工串联的 GUI 流程：

1. `TaggingTab` 读取 Excel `获取列表`，批量执行打标或复用旧结果。
2. 打标完成后进入 `ReviewTab`，由人工逐 case 审核标签和画面描述。
3. 当前 `MainWindow._on_case_approved()` 在“审核通过”时同时做四件事：
   - 根据审核页设备选择重算 `manifest.mode`
   - 立即写 `xlsx`
   - 立即写 `txt`
   - 立即把 case 加入 `ExecutionWorker` 执行队列
4. `ExecutionWorker` 后台执行 `pull -> move -> upload`。

这条链路已经接近目标，但还存在三个明确缺口：

1. `pull` 只能从 ADB 设备拉取，不能优先消费手工从 `/mnt/nvme/CapturedData` 提前拉下来的本地目录。
2. 缺少“全自动模式”。
   - 用户希望在打标界面勾选后，打标结束一进入审核页，后台就立刻开始整批 case 的 `pull/move/upload`。
   - 审核页仍然必须逐条人工审核，但不再负责决定“是否开始执行”。
   - `xlsx` 和 `txt` 不应在进入审核页前统一写入，而应在每条 case 人工审核完成后再补写。
3. 需要确认当前打标是否已经是“压完一个就发千问”的流水线。

经代码检查，第三点当前已经有明确定论：

- `video_tagging_assistant/tagging_service.py` 中的 `run_batch_tagging()` 采用两阶段模型。
- 第一阶段先并行压缩整批待打标 case。
- 第二阶段再并行调用 provider 进行 AI 打标。
- 因此当前行为确实是“先整批压缩完，再整批发千问”，不是逐 case 流水线。

用户已明确表示这不是当前性能瓶颈，因此本次设计不改这一点，只把现状写明并保留后续扩展口。

## 2. 目标

本次设计的目标是：

1. 在 GUI / 主流程不大改的前提下，为 `pull` 新增一个可选的本地临时源 `temp_path`。
2. 让 `temp_path` 下的 RK raw 目录能按 case 精确匹配并被消费式 `move` 到目标 case 目录。
3. 在打标页新增全自动模式勾选项。
4. 在全自动模式下，进入审核页后立即启动整批 case 的后台 `pull/move/upload`。
5. 在全自动模式下，审核页仍然逐条人工审核，但审核动作只负责写回 `txt/xlsx`，不再决定是否入执行队列。
6. 在全自动模式下，整批 case 在开跑前一次性锁定同一个 `Dut_info` 设备档案，并据此确定路径和写回字段。
7. 保持非全自动模式的现有主行为不变。
8. 明确本次不改打标侧“整批压缩后再整批送模”的并发模型。

## 3. 非目标

本次设计不包含以下内容：

1. 不把打标改成逐 case 的压缩/送模流水线。
2. 不允许全自动模式下出现“跳过审核”或“审核不通过后回滚已上传数据”的新分支。
3. 不自动删除服务器端已上传目录。
4. 不把 `temp_path` 扩展成模糊匹配、自动猜目录名或多目录聚合。
5. 不重构为全新的多状态复杂前端，只做满足当前需求的最小拆分。
6. 不改变当前 `ExecutionWorker` 的基础执行顺序 `pull -> move -> upload`。

## 4. 已确认的业务决策

以下约束已经在设计讨论中锁定，本次 spec 直接作为实现依据：

### 4.1 `temp_path` 的目录语义

- `temp_path` 中存放的是手工从 `/mnt/nvme/CapturedData` 拉下来的镜像目录。
- 其下按 RK 目录名分子目录，例如 `117/`、`118/`。
- 程序按 case 的 `rk_suffix` 精确匹配 `temp_path/<rk_suffix>/`。
- 不采用“目录里只有一个候选就自动认领”的模糊逻辑。

### 4.2 `temp_path` 的消费方式

- 命中 `temp_path/<rk_suffix>/` 且目录非空时，作为本次 pull 的本地来源。
- 成功使用后应消费式 `move`，而不是复制。
- 命中失败或目录为空时，再回退到 ADB。

### 4.3 全自动模式的审核约束

- 全自动模式进入审核页后，每一条 case 仍然必须人工审核。
- 不考虑“跳过”或“审核不通过”的自动回滚方案。
- 每条 case 人工审核完成后，立刻补写该 case 的 `txt` 和 `xlsx`。

### 4.4 全自动模式的设备选择约束

- 不能等审核页逐 case 选择设备，因为设备信息会影响 `mode` 与上传路径。
- 必须在开跑前一次性选定一个完整的 `Dut_info` 设备档案。
- 该设备档案在整批 case 上统一生效。

### 4.5 失败处理约束

- 后台 `pull/move/upload` 失败时，不阻塞其他 case 继续执行。
- 失败项在执行队列中标红，允许人工重试。
- 审核和写回可与后台执行解耦，不因为别的 case 失败而停住整批审核。

### 4.6 打标并发模型决策

- 当前“整批压缩完，再整批发千问”的实现已经确认。
- 本次不修改这一行为。

## 5. 方案比较与推荐

讨论中考虑过三类方案：

### 5.1 方案 A：最小改动方案

- `temp_path` 只作为 `pull` 的备用本地源。
- 全自动模式只增加必要的 UI 选项和状态拆分。
- 后台执行与审核写回解耦，但仍复用现有 `MainWindow + ExecutionWorker + ReviewTab` 主结构。
- 打标并发模型保持不变。

优点：

- 改动范围最小。
- 风险最低。
- 能完整满足本次已确认需求。

缺点：

- 状态展示仍基于现有界面，运行态可视化不会一步到位。

### 5.2 方案 B：中等改动方案

- 在方案 A 基础上，把“执行状态”和“审核写回状态”正式拆成两套 UI 展示。
- 明确持久化更多细粒度状态。

优点：

- 语义更清晰，后续扩展更稳。

缺点：

- 超出本次“尽量少改”的目标。

### 5.3 方案 C：顺手改打标流水线

- 在方案 A 或 B 的基础上，把打标也改成“压完一个就发一个”的逐 case 流水线。

优点：

- 理论上减少首批审核前等待时间。

缺点：

- 与当前主诉求不对齐。
- 会显著扩大实现范围。

### 5.4 推荐方案

本次采用 **方案 A：最小改动方案**。

原因：

1. 用户已经确认第 3 点不是主要瓶颈。
2. 本次核心价值在于：
   - 复用手工提前 pull 的本地目录
   - 增加全自动模式
   - 把审核写回和后台执行拆开
3. 这些目标都能在不重构主架构的前提下完成。

## 6. 总体设计

### 6.1 关键设计原则

1. **优先复用当前主 GUI 路径**
   - 以 `video_tagging_assistant/gui/main_window.py` 为实现中心。
   - 不转向旧版 `PipelineController` 路径作为主方案。
2. **把“是否开始执行”从审核动作中拿掉**
   - 这是全自动模式成立的前提。
3. **把设备路径决策前移到打标启动前**
   - 这是全自动模式能提前上传的前提。
4. **在 `pull` 层新增来源选择，而不是复制一套独立执行链**
   - `temp_path` 和 ADB 只应是两种 pull 来源，不应演化成两套后续流程。

### 6.2 新增能力概览

本次新增或调整三条主线：

1. **pull 来源选择**
   - `temp_path` 命中时：本地消费式 `move`
   - 未命中时：原 ADB `pull`

2. **全自动模式**
   - 打标页新增开关
   - 开跑前锁定整批设备
   - 打标结束后直接后台执行整批 case

3. **审核通过后的动作收缩**
   - 从“写回 + 入队执行”改成“仅写回”
   - 执行入队在全自动模式下前移到打标完成后

## 7. 配置设计

### 7.1 新增 `temp_path`

在 `configs/config.json` 顶层新增：

```json
{
  "temp_path": "E:/DV/temp_pull_cache"
}
```

语义如下：

- 类型：字符串
- 默认值：空字符串 `""`
- 含义：本地临时 pull 源根目录
- 关闭方式：留空或缺省

路径规则：

- 其下应按 RK 目录名分子目录，例如：
  - `E:/DV/temp_pull_cache/117/`
  - `E:/DV/temp_pull_cache/118/`
- 每个子目录内容应与设备端 `/mnt/nvme/CapturedData/<rk_suffix>/` 一致。

### 7.2 不新增持久化“自动模式配置”

全自动模式本身是一次运行时选择，而不是需要长期固化的 JSON 配置项。

因此本次不把“是否开启全自动模式”和“本次锁定设备编号”写入 `config.json`，而是作为 GUI 启动前的会话级选择：

- `全自动模式`：由打标页勾选
- `锁定设备档案`：由打标页从 `Dut_info` 读取后选择

这样可以避免把批次级操作状态污染为全局静态配置。

## 8. UI 设计

### 8.1 `TaggingTab` 新增控件

在打标页新增两个控件：

1. `QCheckBox`：`全自动模式`
2. `QComboBox`：`设备编号`

行为如下：

- 默认不勾选 `全自动模式`。
- 未勾选时：
  - 设备下拉框可隐藏或禁用。
  - 当前行为保持不变。
- 勾选后：
  - 必须选择一个来自 `Dut_info` 的完整设备档案。
  - 若未加载工作簿或 `Dut_info` 中没有可选设备，则不允许开始。

### 8.2 `ReviewTab` 在全自动模式下的行为

全自动模式下审核页仍然逐 case 展示 AI 结果，但 UI 行为调整为：

1. 设备信息为只读
   - 不再允许逐 case 改设备。
   - 页面上只展示本批次锁定的设备。
2. `跳过` 按钮禁用或隐藏
   - 因为用户已明确要求全自动模式下每条都必须审核。
3. `通过` 按钮保留
   - 但点击后只触发该 case 的写回，不负责入执行队列。

### 8.3 非全自动模式下的 UI 保持原样

未开启全自动时：

- 审核页仍保留当前逐 case 选设备的能力。
- `跳过` 按钮继续存在。
- `通过` 仍然执行当前模式的主动作。

## 9. 设备锁定与路径决策设计

### 9.1 为什么必须前置锁定设备

当前代码中，审核页选择的设备会影响：

- `manifest.mode`
- `manifest.local_case_root`
- `manifest.server_case_dir`
- `txt` 内容
- `xlsx` 写回字段

而全自动模式要求在进入审核页后就开始后台执行 `pull/move/upload`。

如果设备仍然到审核时逐条决定，就会导致上传路径在执行前不确定。因此必须把设备选择前移。

### 9.2 锁定方式

在全自动模式下，开始打标前一次性选择一个 `Dut_info` 设备档案，并把以下字段锁定为整批通用值：

- `设备编号`
- `模组型号`
- `芯片`
- `采集模式`
- `bit位`
- `帧率`
- `其他信息`

### 9.3 `mode` 的生成规则

沿用当前主窗口逻辑：

```text
mode = {模组型号}_{采集模式}
```

例如：

```text
OV50H40_Action5Pro_DCG HDR
```

该 `mode` 一旦在全自动模式下锁定，就应在整批 manifests 上统一生效，并用于：

- 本地 case 目录
- 服务器 case 目录
- 后续写回路径字段

### 9.4 启动前校验

全自动模式启动前必须校验：

1. 已成功加载工作簿
2. `Dut_info` 至少存在一个设备档案
3. 已选择目标设备档案
4. 选定设备的 `模组型号` 和 `采集模式` 非空

任一不满足时，不允许进入打标。

## 10. pull 来源选择设计

### 10.1 目标

`pull` 对每个 case 只做一件事：把 RK raw 准备到目标 case raw 目录。

区别仅在于来源：

- 来源 A：`temp_path/<rk_suffix>/`
- 来源 B：`adb pull`

后续 `move/upload` 逻辑保持统一。

### 10.2 匹配规则

对某个 case：

1. 从 `manifest.raw_path.name` 或既有 `rk_suffix` 推导目录名，例如 `117`
2. 检查 `Path(temp_path) / rk_suffix`
3. 仅当该路径存在且目录非空时，才认定命中

### 10.3 命中本地临时源时的行为

命中时，执行本地消费式迁移：

1. 统计源目录文件数，作为本次“完整性期望值”
2. 若目标最终目录已存在且文件数已等于源目录文件数：
   - 视为本地最终目录已完整
   - 不再重复迁移
   - 不强行删除临时源目录
   - 记录一条警告日志，提示存在未消费的临时源目录
3. 若目标最终目录不存在或文件数不足：
   - 将临时源目录迁移到最终目录
   - 若最终目录已存在，则按现有 merge 逻辑补齐缺失文件
4. 迁移/合并完成后，用“源目录文件数 == 最终目录文件数”判定成功

这里不再访问 ADB，也不再读取设备端文件数。

### 10.4 未命中本地临时源时的行为

出现以下任一情况时，回退到 ADB：

- `temp_path` 未配置
- `temp_path` 目录不存在
- `temp_path/<rk_suffix>/` 不存在
- `temp_path/<rk_suffix>/` 存在但为空

回退后继续沿用当前 `run_resumable_pull()` 的逻辑：

1. `adb shell find` 统计设备端文件数
2. 如最终目录已完整则跳过
3. 否则重新 `adb pull` 到 `_tmp`
4. merge 到最终目录
5. 用设备端文件数校验最终目录完整性

### 10.5 为什么不做更复杂的冲突处理

如果本地临时源和最终目录同时存在且内容不一致，说明现场状态已经混杂。  
本次不做自动冲突合并策略，只做最小安全规则：

- 优先信任最终目录的“已完整”状态
- 对异常并存状态只告警，不隐式删除多余目录

这样可以避免在自动模式下做高风险清理。

## 11. 执行编排设计

### 11.1 非全自动模式

保持当前主行为：

1. 打标完成
2. 进入审核页
3. 人工审核通过某 case
4. 写 `txt/xlsx`
5. 将该 case 加入执行队列

### 11.2 全自动模式

改为：

1. 打标完成
2. 进入审核页
3. 同时立即将整批 manifests 加入后台执行队列
4. `ExecutionWorker` 开始依次处理：
   - `pull`
   - `move`
   - `upload`
5. 审核页中的人工“通过”只负责当前 case 的写回

### 11.3 为什么选择在打标完成后整批入队

用户已经明确要求：

- 自动模式下，打标界面结束、跳转到审核之后，就直接开始每个 case 的 `pull/upload`

这意味着执行触发点不是“审核通过某一条”，而是“打标批次结束”。

### 11.4 `ExecutionWorker` 的角色保持不变

当前 `ExecutionWorker` 已具备合适的基本模型：

- 前台主队列串行处理 `pull/move`
- 后台上传线程并发处理 `upload`

本次不改其总模型，只扩展其 pull 来源选择能力，并调整何时 `enqueue(manifest)`。

## 12. 审核写回设计

### 12.1 审核动作的职责变化

当前审核页的“通过”动作隐含两件事：

1. 业务审核通过
2. 触发执行

全自动模式下必须把这两件事拆开：

1. 执行已经在后台开始
2. 审核只负责确认和写回

### 12.2 每条 case 的写回时机

用户已明确选择：

- 每审核通过一个 case，就立刻补写这个 case 的 `txt` 和 `xlsx`

因此写回规则是：

1. 用户审核当前 case
2. 点击“通过”
3. 程序写该 case 的 `xlsx`
4. 程序写该 case 的 `txt`
5. 仅在写回全部成功后，前进到下一条 case

### 12.3 审核页的确认式推进

当前 `ReviewTab._handle_pass()` 在 emit 信号后立即 `_advance()`，这不适合新的失败语义。  
本次设计要求把审核页改为“确认式推进”：

1. 审核页发出“请求通过当前 case”的事件
2. 主窗口执行写回逻辑
3. 写回成功后，主窗口显式通知审核页进入下一条
4. 写回失败时，审核页停留在当前 case，不前进

这个机制对非全自动模式同样有价值，因为它也能修复当前“写回失败但页面已前进”的缺陷。

### 12.4 写回失败的行为

若 `xlsx` 或 `txt` 任一步失败：

- 当前 case 保持停留在审核页
- 弹出明确错误或 status bar 错误
- 不自动前进
- 不要求回滚后台已开始的上传

## 13. 状态与错误语义

### 13.1 执行失败

全自动模式下，某 case 的 `pull/move/upload` 失败时：

- 该 case 在执行队列中标红
- 支持沿用现有“重试”入口
- 不阻塞其他 case 继续执行
- 不阻塞当前或后续 case 的人工审核

### 13.2 审核未完成

即使某 case 后台已经上传成功，只要人工还未审核通过：

- `txt/xlsx` 仍未写回
- 该 case 在业务上仍视为“待审核写回”

也就是说，本次设计默认允许“数据归档已完成，但业务台账尚未写回”的短暂中间态。

### 13.3 `temp_path` 不可用

当 `temp_path` 配置非空但不可访问时：

- 不直接中断整批
- 记录日志或界面提示
- 回退到 ADB

这样可以保证该功能作为“优先本地源”而不是“强依赖本地源”存在。

## 16. 本次明确不做的后续优化

为了避免实现期间范围漂移，本次明确不做：

1. 不改成“某个视频压完立即送千问”的打标流水线
2. 不让审核页在全自动模式下支持“驳回后撤销上传”
3. 不引入服务器端半成品自动修复
4. 不引入 `temp_path` 多源优先级或目录扫描智能匹配
5. 不把执行状态和审核写回状态做成新的复杂双面板

## 17. 结论

本次设计的核心是：

1. 为 RK raw `pull` 增加一个可选的本地临时来源 `temp_path`
2. 在命中时用消费式 `move` 取代 ADB
3. 为打标页增加一个真正可用的全自动模式
4. 在全自动模式下，把“后台数据归档执行”与“人工审核写回”拆成两条并行但独立的流程
5. 明确保持当前打标侧“整批压缩后再整批送模”的实现不变

这样可以以最小改动满足当前新增需求，同时避免把不重要的并发优化绑进这次实现范围。
