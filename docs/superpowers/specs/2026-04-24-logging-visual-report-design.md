# 可视化输出与日志收敛设计补充

- 日期：2026-04-24
- 主题：增加终端进度面板、HTML 汇总报告，并将 ffmpeg 等详细日志从终端剥离到日志文件
- 状态：待用户审阅

## 1. 背景

当前工具已经具备：

- 视频扫描
- 视频压缩
- Qwen/DashScope 分析
- 结构化标签输出
- review 文本导出
- 并发压缩与带退避的模型调用

但用户指出了两个新的体验问题：

1. **终端输出噪音太大**
   - 目前运行时会输出大量 `ffmpeg` 压缩日志
   - 这些日志对日常使用意义不大，影响阅读和观察主流程状态

2. **结果可视化不足**
   - 当前主要靠 `review.txt` 看结果
   - 希望有更清晰的汇总方式，便于批量审阅与部署后使用

## 2. 目标

1. 终端不再刷大量 `ffmpeg` 原始输出。
2. 提供一个简洁的终端进度面板/状态输出。
3. 将详细日志写入文件，便于排错。
4. 新增 HTML 汇总报告，用于查看最终结果。
5. 保留现有 `review.txt` 作为轻量审核文本。
6. 将日志和 HTML 报告输出路径配置化。

## 3. 推荐方案

采用“**终端摘要输出 + 文件日志 + HTML 报告**”三层方案。

### 3.1 终端输出层

终端只显示高价值信息：

- 当前总进度
- 当前视频状态
  - 压缩中
  - 分析中
  - 成功
  - 重试中
  - 失败
- 成功/失败数量统计

不直接打印：

- 完整 `ffmpeg` stdout/stderr
- 大段底层原始请求日志

### 3.2 日志层

原始信息写入日志文件，例如：

- `logs/compression/`：压缩日志
- `logs/provider/`：模型请求失败与重试日志
- `logs/run.log`：整体运行日志

终端只给出简洁摘要，详细排错时再看日志文件。

### 3.3 可视化报告层

新增一个 HTML 汇总报告，展示：

- 总任务数
- 成功数
- 失败数
- 待审核数
- 每个视频的：
  - 路径
  - 单选字段
  - 多选字段
  - 画面描述
  - 审核状态
  - 使用模型

## 4. 终端进度设计

建议终端输出改为摘要型格式，例如：

- 启动时：
  - 扫描到多少视频
  - 压缩并发数
  - 模型并发数
- 运行中：
  - `[1/20] 压缩: clip01.mp4`
  - `[1/20] 分析: clip01.mp4`
  - `[1/20] 完成: clip01.mp4`
  - `[2/20] 重试(1/3): clip02.mp4`
- 结束时：
  - 总计
  - 成功/失败数
  - review 路径
  - HTML 报告路径
  - 日志目录

## 5. HTML 报告设计

HTML 报告建议包含：

### 5.1 顶部汇总区

- 总视频数
- 成功数
- 失败数
- 待审核数
- 运行时间（可选）

### 5.2 结果列表区

每条结果展示：

- 视频路径
- 安装方式
- 运动模式
- 运镜方式
- 光源
- 画面特征
- 影像表达
- 画面描述
- 审核状态
- 模型

### 5.3 样式原则

- 纯本地 HTML
- 不依赖外部前端框架
- 可直接双击打开
- 以可读性为主，不追求复杂交互

## 6. 日志设计

### 6.1 压缩日志

每个视频一个压缩日志文件，或统一汇总日志。

记录：

- ffmpeg 命令
- 压缩 stdout/stderr
- 输出文件路径
- 输出文件大小

### 6.2 Provider 日志

记录：

- 请求开始/结束
- 重试次数
- 错误信息
- 最终状态

### 6.3 主运行日志

记录：

- 启动配置摘要
- 任务阶段切换
- 成功/失败统计

## 7. 配置扩展建议

建议新增：

```json
{
  "logging": {
    "log_dir": "output/logs",
    "capture_ffmpeg_output": true,
    "quiet_terminal": true
  },
  "reporting": {
    "generate_html_report": true,
    "html_report_file": "output/report/index.html"
  }
}
```

其中：

- `capture_ffmpeg_output`：是否将 ffmpeg 详细输出写入日志
- `quiet_terminal`：是否压缩终端输出
- `generate_html_report`：是否生成 HTML 报告
- `html_report_file`：HTML 报告输出位置

## 8. 系统改动点

### 8.1 compressor.py

- 增加 quiet/log 参数
- `ffmpeg` 输出重定向到日志文件或静默处理

### 8.2 orchestrator.py

- 增加统一进度输出
- 汇总成功/失败统计
- 结束时打印关键路径

### 8.3 review_exporter.py

- 保留文本导出
- 新增 HTML 导出能力，或拆分为新的 HTML exporter

### 8.4 配置文件

- 增加 logging/reporting 配置段

## 9. 风险与约束

1. 如果日志完全静默，排错时可能不够方便，因此必须保留文件日志。
2. HTML 报告若包含全部结果，视频数量很多时页面会较长，但第一版可接受。
3. 终端“进度面板”第一版建议做成文本摘要，不做复杂动态 UI，以减少实现成本。

## 10. 验收标准

满足以下条件即可认为这一轮扩展可用：

1. 终端不再被大量 ffmpeg 输出刷屏。
2. 压缩和分析阶段有清晰的摘要状态输出。
3. 原始 ffmpeg 和错误信息能在日志文件中查看。
4. 生成一个可直接打开的 HTML 汇总报告。
5. 日志与报告输出位置可通过配置修改。
